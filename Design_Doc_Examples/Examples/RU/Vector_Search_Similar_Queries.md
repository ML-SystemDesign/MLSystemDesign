## 1. Problem Statement

### Problem Space (Что и Почему)

**Что**: 
- Пользователи веб-сервиса задают вопросы, которые обрабатываются с помощью ИИ
- Многие пользователи задают похожие или идентичные вопросы, что приводит к:
  - **Дублированию запросов к ИИ-сервису** (приоритетная проблема)
  - Высоким затратам на обработку (входные токены расходуются в 10 раз больше и получаются дороже выходных)
  - Упущенным возможностям для перелинковки и улучшения SEO
  - Отсутствию возможности показать пользователю уже существующие ответы
- **Ключевой инсайт**: Обнаружены полные дубликаты (distance = 0.0) в существующих данных, что подтверждает необходимость решения
- **Подозрение на системную ошибку**: Есть подозрение, что система может отправлять один и тот же запрос дважды, что приводит к дублированию. Анализ ID найденных дубликатов поможет определить их природу и выявить возможную системную проблему

**Почему**: 
- Текущая система не имеет механизма для определения похожих запросов
- Каждый запрос обрабатывается независимо, даже если идентичный вопрос уже был задан ранее
- Нет связи между новыми и существующими вопросами/ответами на сайте
- Высокая стоимость обработки повторяющихся запросов через ИИ-сервис
- **Важно**: Первичная задача - избавиться от дублированных запросов, которые уже заданы и опубликованы
- Это можно решить **оффлайн** без дополнительной инфраструктуры, все необходимое уже есть on-premises

**Кто затронут**: 
- **Пользователи**: Получают одинаковые ответы на похожие вопросы, тратят время на ожидание обработки
- **Бизнес**: Несет высокие затраты на обработку дублирующихся запросов, упускает возможности для SEO и монетизации
- **Разработчики**: Нужно реализовать систему поиска похожих запросов с учетом ограничений по ресурсам

**Границы проблемы**: 
- Поддержка мультимодальных запросов (текст + изображения) в едином векторном пространстве
- Поиск похожих запросов, а не генерация новых ответов
- Интеграция с существующей системой обработки ИИ-запросов
- Работа с запросами, хранящимися в MongoDB (все существующие запросы)

## 2. Solution Space (Как)

### Возможные подходы

1. **Approach 1: Self-hosted DeepEncoder + FAISS**
   - Pros: 
     - Полный контроль над инфраструктурой
     - Нет зависимости от внешних сервисов
     - Возможность оптимизации под конкретные задачи
     - Низкие операционные затраты при наличии собственных GPU
   - Cons: 
     - Требует GPU-инфраструктуры (недоступна до нового года)
     - Необходимость поддержки и обслуживания
     - Более сложная настройка и развертывание
     - Нужна команда для поддержки инфраструктуры

2. **Approach 2: Elasticsearch с LocalAI**
   - Pros: 
     - Готовая векторная база данных с поддержкой поиска
     - LocalAI поддерживает GPU-ускорение
     - Совместимость с OpenAI API (легкая интеграция)
     - Возможность работы без GPU (CPU fallback)
     - Хорошая документация и сообщество
   - Cons: 
     - Требует настройки LocalAI сервера
     - При работе на CPU может не справиться с потоком запросов
     - Дополнительная инфраструктура для Elasticsearch

3. **Approach 3: ClickHouse с мультимодальными эмбеддингами**
   - Pros: 
     - ClickHouse уже есть в инфраструктуре
     - Поддержка векторного поиска (Array(Float32) для эмбеддингов)
     - Высокая производительность для аналитических запросов
     - Нет необходимости в дополнительной инфраструктуре
     - Хорошая интеграция с существующими системами
   - Cons: 
     - Векторный поиск в ClickHouse менее оптимизирован чем в специализированных векторных БД
     - Может потребоваться дополнительная настройка для векторного поиска через cosineDistance
     - Необходимость выбора мультимодальной модели эмбеддингов

4. **Approach 4: Облачное решение для инференса**
   - Pros: 
     - Быстрый старт без настройки инфраструктуры
     - Масштабируемость из коробки
     - Профессиональная поддержка
     - Оплата по использованию
   - Cons: 
     - Постоянные затраты на использование
     - Зависимость от внешнего провайдера
     - Необходимо проверить стоимость аренды
     - Возможные проблемы с latency при использовании облачных сервисов

5. **Approach 5: CPU-based модели (BLIP, SigLIP) без GPU**
   - Pros: 
     - Можно запустить немедленно без GPU
     - Нет необходимости в дополнительном оборудовании
   - Cons: 
     - Высокая latency (может не уложиться в < 1 секунду)
     - Возможные проблемы с пропускной способностью при высоком потоке запросов
     - Менее точные эмбеддинги по сравнению с GPU-ускоренными моделями

### Выбранный подход
- **Решение**: **jina-v4 + ClickHouse** (выбрано и реализовано как POC)
  - Предобученная модель jina-v4 для генерации мультимодальных эмбеддингов
  - Хранение эмбеддингов прямо в ClickHouse (Array(Float32))
  - Все работает локально on-premises, без дополнительной инфраструктуры
- **Обоснование**:
  - ClickHouse уже есть в инфраструктуре - быстрое развертывание
  - jina-v4 обеспечивает единое векторное пространство для текста и изображений
  - Использование готового представления модели (без модификации векторов)
  - POC уже работает локально, тест на 500k эмбеддингов запланирован на следующую неделю
  - Все необходимые компоненты доступны on-premises

### Trade-offs
- **Оффлайн vs Онлайн**: Оффлайн обработка позволяет избавиться от дубликатов без дополнительной инфраструктуры, но не дает realtime поиск
- **ClickHouse vs Специализированные векторные БД**: ClickHouse менее оптимизирован для векторного поиска, но уже есть в инфраструктуре и не требует дополнительных затрат
- **Готовые эмбеддинги vs Модификация**: Использование готового представления модели проще и быстрее, но может быть менее оптимально для специфических задач
- **On-premises vs Cloud**: On-premises решение дает полный контроль и не требует облачных затрат, но требует поддержки инфраструктуры

## 3. Requirements

### Функциональные требования (ЧТО)

**Приоритет 1: Оффлайн обработка дубликатов**
- [x] Извлечение запросов из MongoDB (начиная с данных за месяц для анализа)
- [x] Генерация эмбеддингов для всех запросов через jina-v4
- [x] Поиск похожих запросов (включая полные дубликаты с distance = 0.0)
- [ ] Анализ статистики по типам запросов (текст/картинка/оба)
- [ ] Оценка качества поиска похожих запросов
- [ ] Измерение времени генерации эмбеддингов
- [x] **Анализ природы дубликатов**: Определение количества дубликатов и их природы через анализ ID найденных дубликатов
  - ✅ Обнаружено 3% дубликатов в тестовой подборке (1000 запросов)
  - ✅ Выявлены паттерны: одновременные (site/web), повторные через минуты (android), тройные
  - 🚧 Анализ продолжается (тройные дубликаты требуют дополнительного изучения)
- [ ] **Диагностика системной ошибки**: Выявление возможной системной проблемы, когда один и тот же запрос отправляется дважды

**Приоритет 2: Интеграция в production (будущее)**
- [ ] При вводе пользовательского запроса (текст + изображения) система должна находить похожие существующие запросы
- [ ] Поддержка мультимодальных запросов: текст и изображения в едином векторном пространстве
- [ ] Отображать список из N похожих вопросов (рекомендуется 3-5) с возможностью выбора
- [ ] Пользователь может выбрать один из похожих вопросов или отправить свой запрос на обработку ИИ
- [ ] Обновление индекса при поступлении новых запросов (near-realtime)
- [ ] Возможность показа рекламного блока перед отправкой собственного запроса (опционально)

### Не функциональные требования (КАК)
- [ ] **Latency**: Время ответа системы поиска похожих вопросов должно быть < 1 секунды на всех платформах (Web mobile, Web PC, Android, iOS)
- [ ] **Производительность**: Система должна обрабатывать запросы с достаточной пропускной способностью для текущего и прогнозируемого трафика
- [ ] **Точность**: Похожие вопросы должны быть релевантными (метрика: precision@K, где K=5)
- [ ] **Масштабируемость**: Система должна масштабироваться при росте количества индексированных вопросов
- [ ] **Надежность**: Fallback механизм при недоступности сервиса поиска (показывать стандартный интерфейс без похожих вопросов)
- [ ] **Мониторинг**: Отслеживание метрик производительности, latency, точности поиска

### Ограничения
- **Технические**: 
  - Доступны GPU: 3080Ti (используется для POC), 3060 Ti (тестировалась)
  - ✅ GPU 3080Ti: производительность до 200 мс, укладываемся в требования
  - ⚠️ GPU 3060 Ti: самые долгие запросы превышают секунду, не укладываемся в требования
  - **Вывод**: Для production требуется GPU уровня 3080Ti или выше
  - Хранение эмбеддингов в ClickHouse (не специализированная векторная БД)
  - Поиск в ClickHouse медленнее чем в специализированных векторных БД, но приемлем для текущих требований
  - Интеграция с существующей системой обработки ИИ-запросов
- **Бизнес**: 
  - Приоритет на оффлайн обработку дубликатов (без дополнительной инфраструктуры)
  - Необходимость экономии на токенах ИИ-сервиса
  - Возможность монетизации через рекламу (будущее)
- **Временные**: 
  - POC работает локально
  - Тест на 500k эмбеддингов запланирован на следующую неделю
- **Ресурсные**: 
  - Все необходимое доступно on-premises
  - ✅ GPU 3080Ti: производительность до 200 мс на генерацию эмбеддинга, укладываемся в требования
  - ⚠️ GPU 3060 Ti: самые долгие запросы превышают секунду, не укладываемся в требования на некоторых запросах
  - Требования к памяти GPU: ~8 ГБ (еле влезли в доступную память GPU)
  - Время генерации зависит от размера и сложности изображений

## 4. System Design

### Архитектура

Система состоит из следующих основных компонентов:

1. **Embedding Service (jina-v4)** - генерация мультимодальных эмбеддингов (text + image) в едином векторном пространстве
2. **Vector Database (ClickHouse)** - хранение эмбеддингов в Array(Float32) и поиск по векторным представлениям
3. **MongoDB** - хранение всех существующих запросов (источник данных: ~2.5M запросов)
4. **Indexing Service** - оффлайн индексация запросов из MongoDB в ClickHouse
5. **Similarity Search Service** - поиск похожих запросов (оффлайн для начала, онлайн в будущем)
6. **Frontend Integration** - интеграция с веб и мобильными приложениями (будущее)

```
┌─────────────────────────────────┐
│   Frontend                      │
│  (Web Mobile/PC, Android, iOS)  │
└──────────────┬──────────────────┘
               │
               │ User Query (text + image)
               ▼
┌──────────────────────────────────┐
│   Similarity Search API          │
│   (принимает запрос с API)       │
└──────────────┬───────────────────┘
               │
               │ 1. Генерация эмбеддинга
               ▼
┌──────────────────────────────────┐
│   jina-v4 Embedding Service      │
│  (Multimodal Embedding)          │
│  text + image → единый вектор    │
│  GPU 3080Ti: до 200ms            │
│  GPU 3060 Ti: > 1000ms (⚠️)     │
│  ~8GB GPU memory                 │
│  Готовое представление модели    │
└──────────────┬───────────────────┘
               │
               │ 2. Поиск похожих
               ▼
┌──────────────────────────────────┐
│  Vector Database (ClickHouse)    │
│  Array(Float32) для эмбеддингов │
│  ~2.5M запросов индексировано    │
└──────────────┬───────────────────┘
               │
               │ 3. Возврат похожих запросов
               │
               ▼
┌──────────────────────────────────┐
│   MongoDB                        │
│   (все существующие запросы)     │
└──────────────────────────────────┘
               ▲
               │
               │ Индексация новых запросов
               │
┌──────────────┴───────────────────┐
│   Indexing Service               │
│   (синхронизация MongoDB →       │
│    Vector DB)                    │
└──────────────────────────────────┘
```

### Компоненты

1. **Embedding Service (jina-v4)**
   - Предобученная модель jina-v4 для единого векторного пространства
   - Поддержка мультимодальных запросов (text + image)
   - Использование готового представления модели (без модификации векторов)
   - Развертывание: локально on-premises
   - GPU: 3080Ti (рекомендуется), 3060 Ti (тестировалась, есть проблемы)
   - **Реальная производительность на GPU 3080Ti (на основе тестирования)**:
     - Генерация эмбеддинга: до 200 мс (в зависимости от конкретных картинок)
     - Время генерации зависит от размера и сложности изображений
     - Текст: обрабатывается быстро
     - **Важно**: Укладываемся в требования (< 1 секунды)
   - **Реальная производительность на GPU 3060 Ti (на основе тестирования)**:
     - ⚠️ Самые долгие запросы превышают секунду (> 1000 мс)
     - Не укладываемся в требования на некоторых запросах
     - **Требуется**: Оптимизация или использование более мощной GPU для production
   - **Требования к памяти GPU**: ~8 ГБ (еле влезли в доступную память GPU)
   - Загрузка GPU: равномерная, около максимальной загрузки памяти и ядер
   - Генерация эмбеддингов для текста и изображений в едином пространстве

2. **Vector Database (ClickHouse)**
   - Использование Array(Float32) для хранения эмбеддингов
   - Векторный поиск через distance функции (L2, cosine)
   - Интеграция с существующей инфраструктурой (on-premises)
   - Индексация запросов с метаданными (ID из MongoDB, текст, изображения, дата, URL)
   - **Текущий статус**: POC работает локально, тест на 500k эмбеддингов запланирован
   - **Ограничение**: Поиск медленнее чем в специализированных векторных БД, но приемлем для текущих требований

3. **MongoDB**
   - Хранилище всех существующих запросов пользователей
   - Источник данных для индексации
   - **Статистика данных**:
     - Всего запросов: ~2,500,395
     - С 1 картинкой: 2,196,183
     - С 2 картинками: 32,230
     - С пустым текстом и 1 картинкой: 1,976,413
     - С пустым текстом и 2 картинками: 26,097
     - С не пустым текстом и 1 картинкой: 219,675
     - С не пустым текстом и 2 картинками: 6,133
   - Метаданные запросов: текст, изображения, дата создания, статус

4. **Indexing Service (Оффлайн)**
   - Извлечение запросов из MongoDB (начиная с данных за месяц для анализа)
   - Генерация мультимодальных эмбеддингов для всех запросов через jina-v4
   - Загрузка эмбеддингов в ClickHouse (Array(Float32))
   - Обработка различных типов запросов (только текст, только изображение, текст+изображение)
   - **Текущий статус**: Smoke test на ~1000 запросах пройден, обнаружены полные дубликаты (distance = 0.0)
   - **Планы**: Тест на 500k эмбеддингов на следующей неделе

5. **Similarity Search Service**
   - **Текущая фаза**: Оффлайн поиск похожих запросов в индексированных данных
   - Поиск похожих запросов по категориям (текст/картинка/оба)
   - Оценка качества поиска (включая обнаружение полных дубликатов с distance = 0.0)
   - **Реальная производительность**: до 200 мс на генерацию эмбеддинга (зависит от конкретных картинок)
   - **Важно**: Даже с задержкой до 200 мс укладываемся в требования
   - **Требования к памяти GPU**: ~8 ГБ (еле влезли в доступную память GPU)
   - **Будущее**: REST API endpoint для realtime поиска похожих вопросов (требует оптимизации для соблюдения < 1s latency)
   - **Будущее**: Принимает мультимодальный запрос (текст + изображения) от frontend
   - **Будущее**: Выполняет поиск похожих векторов через cosineDistance в ClickHouse и возвращает топ-K похожих запросов

6. **Frontend Integration**
   - **Web (Mobile + PC)**: Компонент для отображения похожих вопросов
   - **Android**: Нативная интеграция с API
   - **iOS**: Нативная интеграция с API
   - Обработка выбора пользователя
   - Fallback при недоступности сервиса
   - Поддержка загрузки изображений для мультимодальных запросов

### Data Flow

**Индексация (Оффлайн - текущая фаза)**:
1. Indexing Service извлекает запросы из MongoDB (начиная с данных за месяц)
2. Для каждого запроса (текст + изображения, если есть) генерируется мультимодальный эмбеддинг через jina-v4:
   - **Реальная производительность**: до 200 мс на генерацию эмбеддинга (зависит от конкретных картинок)
   - Время генерации зависит от размера и сложности изображений
   - Текст: обрабатывается быстро
   - **Важно**: Даже с задержкой до 200 мс укладываемся в требования
   - Требования к памяти GPU: ~8 ГБ (еле влезли в доступную память GPU)
3. Используется готовое представление модели (без модификации векторов)
4. Эмбеддинг + метаданные (ID из MongoDB, текст, изображения, дата) сохраняются в ClickHouse (Array(Float32))
5. Связь между ClickHouse и MongoDB поддерживается через ID запроса
6. **Статус**: ✅ Тестовая подборка из 1000 запросов отработала успешно
   - Обнаружено 3% дубликатов (30 запросов)
   - Выявлены паттерны: одновременные (site/web - системная ошибка), повторные через минуты (android - пользовательское поведение), тройные (анализ продолжается)

**Поиск похожих (Оффлайн - текущая фаза)**:
1. Анализ индексированных запросов по категориям (текст/картинка/оба)
2. Поиск похожих запросов в ClickHouse через distance функции
3. Оценка качества поиска (включая обнаружение полных дубликатов с distance = 0.0)
4. Измерение времени генерации эмбеддингов
5. Статистика по типам запросов и их распределению

**Поиск похожих (Онлайн - будущее)**:
1. Пользователь вводит запрос (текст + изображения, опционально) в интерфейсе (Web/Android/iOS)
2. Frontend отправляет мультимодальный запрос в Similarity Search API
3. API отправляет запрос (текст + изображения) в jina-v4 для генерации мультимодального эмбеддинга:
   - **Реальная производительность**: до 200 мс на генерацию эмбеддинга (зависит от конкретных картинок)
   - Даже с задержкой до 200 мс укладываемся в требования (< 1 секунды)
4. API выполняет поиск похожих векторов в ClickHouse через cosineDistance по сгенерированному эмбеддингу
5. API получает ID похожих запросов и загружает полные данные из MongoDB
6. API возвращает топ-K похожих запросов с метаданными (текст, изображения, URL)
7. Frontend отображает список похожих вопросов пользователю
8. Пользователь выбирает один из вопросов или отправляет свой запрос на обработку ИИ

### Метрики и оценка

**Метрики производительности модели**:

- **MRR (Mean Reciprocal Rank)**: Средний обратный ранг первого релевантного результата

**Метрики системы**:
- **Latency P50/P95/P99**: Время ответа системы поиска
- **Throughput**: Количество запросов в секунду
- **Hit Rate**: Процент запросов, где пользователь выбрал похожий вопрос
- **Cost Savings**: Экономия на токенах ИИ-сервиса

## 6. Risks and Mitigation

### Технические риски

- **Риск 1: Высокая latency при генерации эмбеддингов**
  - **Вероятность**: Низкая (реальная производительность: до 200 мс на эмбеддинг)
  - **Влияние**: Низкое (даже с задержкой до 200 мс укладываемся в требования)
  - **Митигация**: 
    - jina-v4 на GPU 3080Ti показывает до 200 мс на генерацию эмбеддинга (зависит от картинок)
    - Время генерации зависит от размера и сложности изображений
    - Текст обрабатывается быстро
    - GPU загружена равномерно, около максимальной загрузки памяти и ядер
    - Для оффлайн обработки текущая производительность приемлема
    - Даже с задержкой до 200 мс укладываемся в требования
    - При росте нагрузки: батчинг запросов для оптимизации
    - При росте нагрузки: кэширование эмбеддингов популярных запросов
    - При росте нагрузки: горизонтальное масштабирование

- **Риск 2: Недостаточная пропускная способность**
  - **Вероятность**: Средняя (на GPU 3060 Ti производительность ниже)
  - **Влияние**: Среднее (на 3080Ti укладываемся, на 3060 Ti есть проблемы)
  - **Митигация**: 
    - ✅ jina-v4 на GPU 3080Ti показывает до 200 мс на генерацию эмбеддинга (зависит от картинок)
    - ⚠️ jina-v4 на GPU 3060 Ti: самые долгие запросы превышают секунду
    - Для оффлайн обработки на 3080Ti текущая производительность приемлема
    - **Для production**: Требуется GPU уровня 3080Ti или выше для гарантированного соблюдения требований
    - GPU загружена равномерно, около максимальной загрузки
    - При росте нагрузки: батчинг запросов для оптимизации производительности
    - При росте нагрузки: горизонтальное масштабирование jina-v4 инстансов
    - При росте нагрузки: использование очередей для обработки запросов
    - При росте нагрузки: rate limiting и graceful degradation

- **Риск 3: Низкое качество поиска (нерелевантные результаты)**
  - **Вероятность**: Низкая (уже обнаружены полные дубликаты с distance = 0.0)
  - **Влияние**: Среднее (smoke test показал наличие дубликатов)
  - **Митигация**: 
    - jina-v4 показала хорошие результаты на smoke test (~1000 запросов)
    - Обнаружены полные дубликаты (distance = 0.0), что подтверждает эффективность подхода
    - Тест на 500k эмбеддингов на следующей неделе для валидации
    - Оценка качества поиска по категориям (текст/картинка/оба)
    - Настройка порога similarity для фильтрации слабо похожих результатов
    - Итеративное улучшение на основе результатов тестирования

- **Риск 4: Системная ошибка дублирования запросов**
  - **Вероятность**: Средняя (есть подозрение на системную проблему)
  - **Влияние**: Высокое (дублирование приводит к лишним затратам на обработку)
  - **Митигация**: 
    - Анализ ID найденных дубликатов для определения их природы
    - Выявление паттернов дублирования (временные метки, пользователи, сессии)
    - Диагностика системной проблемы на основе анализа данных
    - Исправление системной ошибки после выявления причины
    - Мониторинг дубликатов после исправления

- **Риск 5: Медленный поиск в ClickHouse**
  - **Вероятность**: Средняя (ClickHouse не специализированная векторная БД)
  - **Влияние**: Низкое (текущие требования: 2 запроса/сек легко укладываются)
  - **Митигация**: 
    - ClickHouse обеспечивает приемлемую производительность для текущих требований
    - Оптимизация через индексы и материализованные представления
    - При росте нагрузки: рассмотреть миграцию на специализированную векторную БД
    - Мониторинг производительности поиска

### Бизнес риски

- **Риск 1: Высокая стоимость облачного решения**
  - **Вероятность**: Средняя
  - **Влияние**: Среднее (может превысить бюджет)
  - **Митигация**: 
    - Детальный анализ стоимости различных провайдеров
    - Расчет ROI (экономия токенов vs стоимость инфраструктуры)
    - Начало с self-hosted решения для контроля затрат

- **Риск 2: Пользователи не используют похожие вопросы**
  - **Вероятность**: Средняя
  - **Влияние**: Высокое (фича не принесет ожидаемой пользы)
  - **Митигация**: 
    - UX оптимизация для привлечения внимания к похожим вопросам
    - A/B тестирование различных вариантов отображения
    - Анализ поведения пользователей и итеративное улучшение
    - Возможность показа рекламы для монетизации даже при низком использовании

- **Риск 3: Негативное влияние на SEO из-за дублирования контента**
  - **Вероятность**: Низкая
  - **Влияние**: Среднее
  - **Митигация**: 
    - Правильная настройка canonical URLs
    - Использование noindex для страниц похожих вопросов при необходимости
    - Консультация с SEO специалистами

### Specification Gaming риски

- **Риск: Модель оптимизирует не ту метрику**
  - Модель может находить технически похожие вопросы, но не семантически релевантные
  - **Митигация**: 
    - Использование метрик, которые отражают реальное поведение пользователей (hit rate, click-through rate)
    - Регулярная валидация результатов на реальных данных
    - Фокус на бизнес-метриках (экономия токенов, engagement) а не только на технических метриках модели

- **Риск: Overfitting на популярные запросы**
  - Система может хорошо работать только для частых запросов
  - **Митигация**: 
    - Тестирование на разнообразных запросах (long-tail)
    - Мониторинг качества для разных типов запросов
    - Балансировка между популярными и редкими запросами

## 8. Success Metrics

### Метрики успеха

**Технические метрики**:
- **Latency P95**: < 1 секунда для 95% запросов (целевое значение)
- **Latency P99**: < 1.5 секунды для 99% запросов
- **Throughput**: Обработка минимум X запросов в секунду (определить на основе текущего трафика)
- **Uptime**: > 99.5% доступности сервиса поиска
- **Precision@5**: > 0.7 (70% похожих вопросов должны быть релевантными)

**Бизнес метрики**:
- **Hit Rate**: > 20% пользователей выбирают похожий вопрос вместо отправки своего
- **Cost Savings**: Снижение затрат на токены ИИ-сервиса на 15-25% за счет переиспользования ответов
- **SEO Impact**: Увеличение внутренних переходов на 10-15% (перелинковка)
- **User Engagement**: Увеличение времени на сайте на 5-10% за счет просмотра похожих вопросов

**UX метрики**:
- **Click-through Rate (CTR)**: > 15% пользователей кликают на похожие вопросы
- **Bounce Rate**: Снижение bounce rate на страницах с похожими вопросами
- **User Satisfaction**: Положительный feedback от пользователей (опросы, рейтинги)

### KPI

- **Primary KPI: Экономия на токенах ИИ-сервиса**
  - Метрика: Процент запросов, которые были обработаны через похожие вопросы вместо ИИ
  - Целевое значение: 15-25% экономии на входных токенах
  - Расчет: (Количество выбранных похожих вопросов / Общее количество запросов) × Стоимость обработки одного запроса

- **Secondary KPI: Улучшение SEO через перелинковку**
  - Метрика: Количество внутренних переходов через похожие вопросы
  - Целевое значение: Увеличение на 10-15% по сравнению с baseline
  - Измерение: Google Analytics / внутренняя аналитика

- **Tertiary KPI: User Engagement**
  - Метрика: Время на сайте, количество просмотренных страниц
  - Целевое значение: Увеличение на 5-10%
  - Измерение: Аналитика поведения пользователей

## 10. Notes and Learnings

### Ключевые инсайты

- **Оффлайн подход**: Начало с оффлайн обработки дубликатов позволяет избавиться от проблемы без дополнительной инфраструктуры, используя существующие on-premises ресурсы.

- **Реальная производительность jina-v4**: 
  - ✅ **GPU 3080Ti**: Модель показывает производительность до 200 мс на генерацию эмбеддинга (зависит от конкретных картинок). Укладываемся в требования (< 1 секунды).
  - ⚠️ **GPU 3060 Ti**: Самые долгие запросы превышают секунду (> 1000 мс). Не укладываемся в требования на некоторых запросах.
  - **Вывод**: Для production требуется GPU уровня 3080Ti или выше для гарантированного соблюдения требований по latency.
  - Требования к памяти GPU: ~8 ГБ (еле влезли в доступную память GPU).

- **Экономия токенов**: Входные токены стоят в 10 раз дороже выходных, поэтому даже небольшая экономия через переиспользование ответов может быть значительной.

- **SEO преимущества**: Перелинковка через похожие вопросы может улучшить ранкинг в поисковых системах, что является дополнительным бизнес-бенефитом.

- **Монетизация**: Возможность показа рекламы перед отправкой собственного запроса создает дополнительный источник дохода.

### Вопросы для дальнейшего изучения

- **Оптимизация jina-v4**: 
  - ✅ На GPU 3080Ti: до 200 мс на эмбеддинг (удовлетворительно)
  - ⚠️ На GPU 3060 Ti: самые долгие запросы превышают секунду (требуется оптимизация)
  - Оптимизация использования памяти GPU (~8 ГБ) для масштабирования на полный объем данных (~2.5M запросов)
  - Рассмотреть батчинг и оптимизацию обработки изображений для 3060 Ti
  - Рассмотреть более легкие модели для менее мощных GPU

- **Стоимость облачных решений**: Детальный анализ стоимости различных провайдеров (OpenAI Embeddings API, Cohere, Hugging Face Inference API) vs self-hosted.

- **Масштабирование jina-v4**: 
  - Оптимизация работы jina-v4 на GPU 3080Ti с учетом реальной производительности (до 200 мс на эмбеддинг)
  - ⚠️ Проблема производительности на GPU 3060 Ti (самые долгие запросы > 1 сек) требует решения
  - Требования к памяти GPU (~8 ГБ) для обработки полного объема данных
  - Рассмотреть батчинг, оптимизацию размера изображений
  - Для менее мощных GPU: рассмотреть более легкие модели или оптимизацию pipeline
  - Возможность горизонтального масштабирования при росте нагрузки

- **Кэширование стратегия**: Какие запросы кэшировать? Как долго хранить кэш? Какой размер кэша оптимален?

- **Мультимодальность**: Анализ качества поиска для различных комбинаций (только текст, только изображение, текст+изображение). Статистика показывает преобладание запросов с изображениями (87.8% с 1 картинкой).

- **Оптимизация поиска в ClickHouse**: Дальнейшая оптимизация векторного поиска в ClickHouse для улучшения производительности при росте объема данных. Рассмотреть специализированные векторные БД при необходимости.

- **Мультиязычность**: Поддержка поиска похожих вопросов на разных языках. Требуются ли отдельные модели для каждого языка?

- **Персонализация**: Можно ли улучшить релевантность, учитывая историю запросов пользователя или его предпочтения?

- **Обратная связь**: Механизм сбора feedback от пользователей о релевантности похожих вопросов для улучшения модели.

- **Диагностика системной ошибки**: ✅ Частично завершено
  - Обнаружено 3% дубликатов в тестовой подборке (1000 запросов)
  - Выявлены паттерны дублирования:
    - **Одновременные дубликаты (site/web)**: Вероятная системная ошибка при отправке
    - **Повторные запросы через минуты (android)**: Пользовательское поведение (не получил ответ после рекламы)
    - **Тройные запросы**: Требуют дополнительного анализа
  - **Следующие шаги**: 
    - Детальный анализ тройных дубликатов
    - Исследование причин одновременных дубликатов для исправления системной ошибки
