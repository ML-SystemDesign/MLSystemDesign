## 1. Problem Statement

### Problem Space (Что и Почему)

**Что**: 
- Пользователи веб-сервиса задают вопросы, которые обрабатываются с помощью ИИ
- Многие пользователи задают похожие или идентичные вопросы, что приводит к:
  - Дублированию запросов к ИИ-сервису
  - Высоким затратам на обработку (входные токены расходюутся в 10 раз больше и получаются дороже выходных)
  - Упущенным возможностям для перелинковки и улучшения SEO
  - Отсутствию возможности показать пользователю уже существующие ответы

**Почему**: 
- Текущая система не имеет механизма для определения похожих запросов (в near-realtime режиме)
- Каждый запрос обрабатывается независимо, даже если идентичный вопрос уже был задан ранее
- Нет связи между новыми и существующими вопросами/ответами на сайте
- Высокая стоимость обработки повторяющихся запросов через ИИ-сервис

**Кто затронут**: 
- **Пользователи**: Получают одинаковые ответы на похожие вопросы, тратят время на ожидание обработки
- **Бизнес**: Несет высокие затраты на обработку дублирующихся запросов, упускает возможности для SEO и монетизации
- **Разработчики**: Нужно реализовать систему поиска похожих запросов с учетом ограничений по ресурсам

**Границы проблемы**: 
- Поддержка мультимодальных запросов (текст + изображения) в едином векторном пространстве
- Поиск похожих запросов, а не генерация новых ответов
- Интеграция с существующей системой обработки ИИ-запросов
- Работа с запросами, хранящимися в MongoDB (все существующие запросы)

## 2. Solution Space (Как)

### Возможные подходы

1. **Approach 1: Self-hosted DeepEncoder + FAISS**
   - Pros: 
     - Полный контроль над инфраструктурой
     - Нет зависимости от внешних сервисов
     - Возможность оптимизации под конкретные задачи
     - Низкие операционные затраты при наличии собственных GPU
   - Cons: 
     - Требует GPU-инфраструктуры (недоступна до нового года)
     - Необходимость поддержки и обслуживания
     - Более сложная настройка и развертывание
     - Нужна команда для поддержки инфраструктуры

2. **Approach 2: Elasticsearch с LocalAI**
   - Pros: 
     - Готовая векторная база данных с поддержкой поиска
     - LocalAI поддерживает GPU-ускорение
     - Совместимость с OpenAI API (легкая интеграция)
     - Возможность работы без GPU (CPU fallback)
     - Хорошая документация и сообщество
   - Cons: 
     - Требует настройки LocalAI сервера
     - При работе на CPU может не справиться с потоком запросов
     - Дополнительная инфраструктура для Elasticsearch

3. **Approach 3: ClickHouse с мультимодальными эмбеддингами**
   - Pros: 
     - ClickHouse уже есть в инфраструктуре
     - Поддержка векторного поиска (Array(Float32) для эмбеддингов)
     - Высокая производительность для аналитических запросов
     - Нет необходимости в дополнительной инфраструктуре
     - Хорошая интеграция с существующими системами
   - Cons: 
     - Векторный поиск в ClickHouse менее оптимизирован чем в специализированных векторных БД
     - Может потребоваться дополнительная настройка для kNN поиска
     - Необходимость выбора мультимодальной модели эмбеддингов

4. **Approach 4: Облачное решение для инференса**
   - Pros: 
     - Быстрый старт без настройки инфраструктуры
     - Масштабируемость из коробки
     - Профессиональная поддержка
     - Оплата по использованию
   - Cons: 
     - Постоянные затраты на использование
     - Зависимость от внешнего провайдера
     - Необходимо проверить стоимость аренды
     - Возможные проблемы с latency при использовании облачных сервисов

4. **Approach 4: CPU-based модели (BLIP, SigLIP) без GPU**
   - Pros: 
     - Можно запустить немедленно без GPU
     - Нет необходимости в дополнительном оборудовании
   - Cons: 
     - Высокая latency (может не уложиться в < 1 секунду)
     - Возможные проблемы с пропускной способностью при высоком потоке запросов
     - Менее точные эмбеддинги по сравнению с GPU-ускоренными моделями

### Выбранный подход
- **Гибридный подход с оценкой вариантов**: 
  - **Вариант A**: ClickHouse (уже в инфраструктуре) + LocalAI с мультимодальной моделью
  - **Вариант B**: Elasticsearch + LocalAI с мультимодальной моделью
  - Требуется сравнение производительности и удобства интеграции
- **Обоснование**:
  - ClickHouse уже есть в инфраструктуре - это может ускорить развертывание
  - Необходима поддержка мультимодальности (text + image) в едином векторном пространстве
  - MVP нужно реализовать за 3 недели, GPU недоступны до нового года
  - Можно протестировать оба варианта и выбрать оптимальный
  - Параллельно изучить стоимость облачных решений как альтернативу

### Trade-offs
- **Latency vs Cost**: CPU решение дешевле, но медленнее. Нужно найти баланс между стоимостью и производительностью
- **Self-hosted vs Cloud**: Self-hosted дает контроль, но требует ресурсов. Cloud быстрее в развертывании, но дороже в долгосрочной перспективе
- **Простота vs Производительность**: Простые модели быстрее развернуть, но могут не справиться с нагрузкой. Сложные модели требуют GPU
- **Время разработки vs Оптимальность**: MVP на CPU быстрее, но может потребовать доработки при росте нагрузки

## 3. Requirements

### Функциональные требования (ЧТО)
- [ ] При вводе пользовательского запроса (текст + изображения) система должна находить похожие существующие запросы
- [ ] Поддержка мультимодальных запросов: текст и изображения в едином векторном пространстве
- [ ] Отображать список из N похожих вопросов (рекомендуется 3-5) с возможностью выбора
- [ ] Пользователь может выбрать один из похожих вопросов или отправить свой запрос на обработку ИИ
- [ ] Система должна индексировать все существующие запросы из MongoDB в векторной базе данных
- [ ] Обновление индекса при поступлении новых запросов (near-realtime)
- [ ] Возможность показа рекламного блока перед отправкой собственного запроса (опционально)

### Не функциональные требования (КАК)
- [ ] **Latency**: Время ответа системы поиска похожих вопросов должно быть < 1 секунды на всех платформах (Web mobile, Web PC, Android, iOS)
- [ ] **Производительность**: Система должна обрабатывать запросы с достаточной пропускной способностью для текущего и прогнозируемого трафика
- [ ] **Точность**: Похожие вопросы должны быть релевантными (метрика: precision@K, где K=5)
- [ ] **Масштабируемость**: Система должна масштабироваться при росте количества индексированных вопросов
- [ ] **Надежность**: Fallback механизм при недоступности сервиса поиска (показывать стандартный интерфейс без похожих вопросов)
- [ ] **Мониторинг**: Отслеживание метрик производительности, latency, точности поиска

### Ограничения
- **Технические**: 
  - Нет доступа к GPU серверам до нового года (2025)
  - Доступны GPU: 3080, 3060Ti (можно использовать для инференса)
  - Необходимость работы на CPU для MVP
  - Интеграция с существующей системой обработки ИИ-запросов
- **Бизнес**: 
  - Бюджет на облачные решения требует проверки
  - Необходимость экономии на токенах ИИ-сервиса
  - Возможность монетизации через рекламу
- **Временные**: 
  - MVP должен быть готов за 3 недели
  - GPU инфраструктура будет доступна после нового года
- **Ресурсные**: 
  - Ограниченные вычислительные ресурсы без GPU
  - Необходимость оптимизации для работы на CPU или простых GPU

## 4. System Design

### Архитектура

Система состоит из следующих основных компонентов:

1. **Embedding Service** (LocalAI) - сервис для генерации мультимодальных эмбеддингов (text + image)
2. **Vector Database** (ClickHouse или Elasticsearch) - хранение и поиск по векторным представлениям
3. **MongoDB** - хранение всех существующих запросов (источник данных)
4. **Indexing Service** - индексация запросов из MongoDB в векторную БД
5. **Similarity Search API** - API для поиска похожих вопросов
6. **Frontend Integration** - интеграция с веб и мобильными приложениями

```
┌─────────────────────────────────┐
│   Frontend                      │
│  (Web Mobile/PC, Android, iOS)  │
└──────────────┬──────────────────┘
               │
               │ User Query (text + image)
               ▼
┌──────────────────────────────────┐
│   Similarity Search API          │
│   (принимает запрос с API)       │
└──────────────┬───────────────────┘
               │
               │ 1. Генерация эмбеддинга
               ▼
┌──────────────────────────────────┐
│   jina-v4 Embedding Service      │
│  (Multimodal Embedding)          │
│  text + image → единый вектор    │
│  < 0.1s, 6-7GB RAM               │
└──────────────┬───────────────────┘
               │
               │ 2. Поиск похожих
               ▼
┌──────────────────────────────────┐
│  Vector Database                 │
│  (ClickHouse / Elasticsearch)    │
│  (хранит эмбеддинги запросов)    │
└──────────────┬───────────────────┘
               │
               │ 3. Возврат похожих запросов
               │
               ▼
┌──────────────────────────────────┐
│   MongoDB                        │
│   (все существующие запросы)     │
└──────────────────────────────────┘
               ▲
               │
               │ Индексация новых запросов
               │
┌──────────────┴───────────────────┐
│   Indexing Service               │
│   (синхронизация MongoDB →       │
│    Vector DB)                    │
└──────────────────────────────────┘
```

### Компоненты

1. **Embedding Service (jina-v4)**
   - Развертывание через Docker или напрямую
   - Поддержка мультимодальных запросов (text + image)
   - Модель: jina-v4 для единого векторного пространства
   - Требования: ~6-7 ГБ RAM (подходит для машин с 4x GPU 3090)
   - Производительность: < 0.1 секунды на генерацию эмбеддинга
   - Генерация эмбеддингов для текста и изображений в едином пространстве

2. **Vector Database (ClickHouse или Elasticsearch)**
   - **Вариант A: ClickHouse**
     - Использование Array(Float32) для хранения эмбеддингов
     - Векторный поиск через distance функции (L2, cosine)
     - Интеграция с существующей инфраструктурой
   - **Вариант B: Elasticsearch**
     - Векторный поиск через dense_vector field
     - Оптимизированный kNN поиск
   - Индексация запросов с метаданными (ID из MongoDB, текст, изображения, дата, URL)

3. **MongoDB**
   - Хранилище всех существующих запросов пользователей
   - Источник данных для индексации
   - Метаданные запросов: текст, изображения, дата создания, статус

4. **Indexing Service**
   - Синхронизация с MongoDB (near-realtime или периодическая)
   - Генерация мультимодальных эмбеддингов для новых запросов через jina-v4 (< 0.1 секунды на запрос)
   - Обновление индекса в векторной БД (ClickHouse/Elasticsearch)
   - Обработка как текстовых, так и мультимодальных запросов

5. **Similarity Search API**
   - REST API endpoint для поиска похожих вопросов
   - Принимает мультимодальный запрос (текст + изображения) от frontend
   - Генерирует эмбеддинг через jina-v4 (единое векторное пространство, < 0.1 секунды)
   - Выполняет kNN поиск в векторной БД
   - Возвращает топ-K похожих запросов с метаданными из MongoDB

6. **Frontend Integration**
   - **Web (Mobile + PC)**: Компонент для отображения похожих вопросов
   - **Android**: Нативная интеграция с API
   - **iOS**: Нативная интеграция с API
   - Обработка выбора пользователя
   - Fallback при недоступности сервиса
   - Поддержка загрузки изображений для мультимодальных запросов

### Data Flow

**Индексация (Offline/Near-realtime)**:
1. Indexing Service мониторит MongoDB на наличие новых запросов
2. Для каждого нового запроса (текст + изображения, если есть) генерируется мультимодальный эмбеддинг через jina-v4 (< 0.1 секунды)
3. Эмбеддинг + метаданные (ID из MongoDB, текст, изображения, дата) сохраняются в векторной БД (ClickHouse/Elasticsearch)
4. Связь между векторной БД и MongoDB поддерживается через ID запроса

**Поиск похожих (Online)**:
1. Пользователь вводит запрос (текст + изображения, опционально) в интерфейсе (Web/Android/iOS)
2. Frontend отправляет мультимодальный запрос в Similarity Search API
3. API отправляет запрос (текст + изображения) в jina-v4 для генерации мультимодального эмбеддинга в едином векторном пространстве (< 0.1 секунды)
4. API выполняет kNN поиск в векторной БД (ClickHouse/Elasticsearch) по сгенерированному эмбеддингу
5. API получает ID похожих запросов и загружает полные данные из MongoDB
6. API возвращает топ-K похожих запросов с метаданными (текст, изображения, URL)
7. Frontend отображает список похожих вопросов пользователю
8. Пользователь выбирает один из вопросов или отправляет свой запрос на обработку ИИ

## 5. ML Components (если применимо)

### Тип задачи
- [x] Другое: **Multimodal Semantic Similarity Search / Information Retrieval**
- Задача поиска похожих мультимодальных запросов (текст + изображения) с использованием векторных представлений в едином пространстве

### Модель и подход

**Модель для генерации мультимодальных эмбеддингов**:
- **Требование**: Модель должна работать с текстом и изображениями в едином векторном пространстве
- **Выбранная модель: jina-v4**
  - Модель jina-v4 обеспечивает генерацию эмбеддингов для текста и изображений в едином векторном пространстве
  - Требования к ресурсам: ~6-7 ГБ RAM (подходит для локальных машин с 4x GPU 3090)
  - Производительность: генерация эмбеддинга занимает менее 0.1 секунды
  - Отлично подходит для near-realtime поиска похожих запросов
- **Альтернативные кандидаты (рассматривались)**:
  - `openai/clip-vit-base-patch32` - легкая CLIP модель
  - `sentence-transformers/clip-ViT-B-32` - оптимизированная версия
  - `google/siglip-base-patch16-224` - SigLIP (более современная альтернатива CLIP)
  - `openai/clip-vit-large-patch14` - более точная CLIP модель
  - `google/siglip-large-patch16-384` - SigLIP large

**Подход**:
- Использование pre-trained мультимодальной модели jina-v4
- Модель обучена на парах текст-изображение для понимания семантической связи
- Генерация dense vector representations в едином пространстве для текста и изображений
- Высокая производительность: < 0.1 секунды на генерацию эмбеддинга
- Умеренные требования к ресурсам: 6-7 ГБ RAM
- Поиск похожих векторов через cosine similarity или L2 distance

**Векторный поиск**:
- **Вариант A: ClickHouse**
  - kNN поиск через distance функции (L2Distance, cosineDistance)
  - Хранение в Array(Float32)
  - Оптимизация через индексы и материализованные представления
- **Вариант B: Elasticsearch**
  - k-Nearest Neighbors (kNN) поиск
  - Использование `dense_vector` field type
  - Индексация через HNSW (Hierarchical Navigable Small World) алгоритм

### Метрики и оценка

**Метрики производительности модели**:
- **Precision@K**: Доля релевантных результатов в топ-K (K=5)
- **Recall@K**: Доля найденных релевантных результатов
- **MRR (Mean Reciprocal Rank)**: Средний обратный ранг первого релевантного результата
- **NDCG@K**: Normalized Discounted Cumulative Gain для оценки качества ранжирования

**Метрики системы**:
- **Latency P50/P95/P99**: Время ответа системы поиска
- **Throughput**: Количество запросов в секунду
- **Hit Rate**: Процент запросов, где пользователь выбрал похожий вопрос
- **Cost Savings**: Экономия на токенах ИИ-сервиса

**Оценка качества**:
- Ручная оценка релевантности на выборке запросов
- A/B тестирование с контрольной группой (без похожих вопросов)
- Мониторинг метрик в production

### Данные

**Источники данных**:
- MongoDB - все существующие запросы пользователей
- Метаданные: ID запроса, текст запроса, изображения (если есть), дата создания, статус, ответ (если есть), URL

**Подготовка данных**:
- Извлечение запросов из MongoDB
- Очистка текста (удаление HTML, нормализация)
- Предобработка изображений (resize, нормализация для модели)
- Генерация мультимодальных эмбеддингов для всех запросов (текст + изображения в едином пространстве)
- Индексация в векторной БД (ClickHouse/Elasticsearch) с метаданными и ссылкой на MongoDB ID

**Обновление данных**:
- Near-realtime или периодическая индексация новых запросов из MongoDB
- Обработка как текстовых, так и мультимодальных запросов
- Периодическая переиндексация при обновлении модели
- Мониторинг актуальности индекса и синхронизации с MongoDB

## 6. Risks and Mitigation

### Технические риски

- **Риск 1: Высокая latency при генерации эмбеддингов**
  - **Вероятность**: Низкая (jina-v4 показывает < 0.1s)
  - **Влияние**: Среднее (требование < 1 секунды должно выполняться)
  - **Митигация**: 
    - jina-v4 обеспечивает < 0.1s на генерацию эмбеддинга
    - Кэширование эмбеддингов популярных запросов
    - Асинхронная генерация эмбеддингов где возможно
    - Параллельная обработка на нескольких GPU (4x 3090)
    - Мониторинг latency в production

- **Риск 2: Недостаточная пропускная способность**
  - **Вероятность**: Средняя
  - **Влияние**: Высокое (система может не справиться с потоком запросов)
  - **Митигация**: 
    - jina-v4 с производительностью < 0.1s позволяет обрабатывать ~10 запросов/сек на один инстанс
    - Горизонтальное масштабирование jina-v4 инстансов
    - Использование очередей для обработки запросов
    - Rate limiting и graceful degradation
    - Мониторинг нагрузки и автоматическое масштабирование
    - Использование всех доступных GPU (4x 3090) для параллельной обработки

- **Риск 3: Низкое качество поиска (нерелевантные результаты)**
  - **Вероятность**: Средняя
  - **Влияние**: Высокое (пользователи не будут использовать фичу)
  - **Митигация**: 
    - Тщательный выбор модели эмбеддингов
    - Тестирование на реальных данных перед запуском
    - Настройка порога similarity для фильтрации слабо похожих результатов
    - A/B тестирование и итеративное улучшение
    - Сбор feedback от пользователей

- **Риск 4: Проблемы с синхронизацией индекса**
  - **Вероятность**: Средняя
  - **Влияние**: Среднее (новые вопросы могут не появляться в поиске)
  - **Митигация**: 
    - Надежный механизм индексации с retry логикой
    - Мониторинг lag между публикацией и индексацией
    - Периодическая проверка целостности индекса

### Бизнес риски

- **Риск 1: Высокая стоимость облачного решения**
  - **Вероятность**: Средняя
  - **Влияние**: Среднее (может превысить бюджет)
  - **Митигация**: 
    - Детальный анализ стоимости различных провайдеров
    - Расчет ROI (экономия токенов vs стоимость инфраструктуры)
    - Начало с self-hosted решения для контроля затрат

- **Риск 2: Пользователи не используют похожие вопросы**
  - **Вероятность**: Средняя
  - **Влияние**: Высокое (фича не принесет ожидаемой пользы)
  - **Митигация**: 
    - UX оптимизация для привлечения внимания к похожим вопросам
    - A/B тестирование различных вариантов отображения
    - Анализ поведения пользователей и итеративное улучшение
    - Возможность показа рекламы для монетизации даже при низком использовании

- **Риск 3: Негативное влияние на SEO из-за дублирования контента**
  - **Вероятность**: Низкая
  - **Влияние**: Среднее
  - **Митигация**: 
    - Правильная настройка canonical URLs
    - Использование noindex для страниц похожих вопросов при необходимости
    - Консультация с SEO специалистами

### Specification Gaming риски

- **Риск: Модель оптимизирует не ту метрику**
  - Модель может находить технически похожие вопросы, но не семантически релевантные
  - **Митигация**: 
    - Использование метрик, которые отражают реальное поведение пользователей (hit rate, click-through rate)
    - Регулярная валидация результатов на реальных данных
    - Фокус на бизнес-метриках (экономия токенов, engagement) а не только на технических метриках модели

- **Риск: Overfitting на популярные запросы**
  - Система может хорошо работать только для частых запросов
  - **Митигация**: 
    - Тестирование на разнообразных запросах (long-tail)
    - Мониторинг качества для разных типов запросов
    - Балансировка между популярными и редкими запросами

## 8. Success Metrics

### Метрики успеха

**Технические метрики**:
- **Latency P95**: < 1 секунда для 95% запросов (целевое значение)
- **Latency P99**: < 1.5 секунды для 99% запросов
- **Throughput**: Обработка минимум X запросов в секунду (определить на основе текущего трафика)
- **Uptime**: > 99.5% доступности сервиса поиска
- **Precision@5**: > 0.7 (70% похожих вопросов должны быть релевантными)

**Бизнес метрики**:
- **Hit Rate**: > 20% пользователей выбирают похожий вопрос вместо отправки своего
- **Cost Savings**: Снижение затрат на токены ИИ-сервиса на 15-25% за счет переиспользования ответов
- **SEO Impact**: Увеличение внутренних переходов на 10-15% (перелинковка)
- **User Engagement**: Увеличение времени на сайте на 5-10% за счет просмотра похожих вопросов

**UX метрики**:
- **Click-through Rate (CTR)**: > 15% пользователей кликают на похожие вопросы
- **Bounce Rate**: Снижение bounce rate на страницах с похожими вопросами
- **User Satisfaction**: Положительный feedback от пользователей (опросы, рейтинги)

### KPI

- **Primary KPI: Экономия на токенах ИИ-сервиса**
  - Метрика: Процент запросов, которые были обработаны через похожие вопросы вместо ИИ
  - Целевое значение: 15-25% экономии на входных токенах
  - Расчет: (Количество выбранных похожих вопросов / Общее количество запросов) × Стоимость обработки одного запроса

- **Secondary KPI: Улучшение SEO через перелинковку**
  - Метрика: Количество внутренних переходов через похожие вопросы
  - Целевое значение: Увеличение на 10-15% по сравнению с baseline
  - Измерение: Google Analytics / внутренняя аналитика

- **Tertiary KPI: User Engagement**
  - Метрика: Время на сайте, количество просмотренных страниц
  - Целевое значение: Увеличение на 5-10%
  - Измерение: Аналитика поведения пользователей

### Оценка эффективности

**Методы оценки**:
1. **A/B тестирование**:
   - Контрольная группа: Пользователи без фичи похожих вопросов
   - Тестовая группа: Пользователи с фичей
   - Период тестирования: 2-4 недели
   - Сравнение метрик между группами

2. **Мониторинг в production**:
   - Real-time дашборды с ключевыми метриками
   - Алерты при отклонении от целевых значений
   - Еженедельные отчеты для стейкхолдеров

3. **Качественный анализ**:
   - Ручная проверка релевантности результатов на выборке
   - Сбор feedback от пользователей
   - Анализ edge cases и проблемных сценариев

4. **Итеративное улучшение**:
   - Еженедельный review метрик
   - Приоритизация улучшений на основе данных
   - Эксперименты с различными моделями и параметрами

## 9. Notes and Learnings

### Ключевые инсайты

- **Важность latency**: Требование < 1 секунды критично для пользовательского опыта. Это может быть вызовом на CPU, но необходимо для принятия фичи пользователями.

- **Гибридный подход**: Начало с CPU решения для MVP позволяет быстро запустить фичу, но требует плана миграции на GPU для масштабирования.

- **Экономия токенов**: Входные токены стоят в 10 раз дороже выходных, поэтому даже небольшая экономия через переиспользование ответов может быть значительной.

- **SEO преимущества**: Перелинковка через похожие вопросы может улучшить ранкинг в поисковых системах, что является дополнительным бизнес-бенефитом.

- **Монетизация**: Возможность показа рекламы перед отправкой собственного запроса создает дополнительный источник дохода.

### Вопросы для дальнейшего изучения

- **Оптимизация jina-v4**: Дальнейшая оптимизация производительности и использования памяти модели jina-v4 для масштабирования

- **Стоимость облачных решений**: Детальный анализ стоимости различных провайдеров (OpenAI Embeddings API, Cohere, Hugging Face Inference API) vs self-hosted.

- **Масштабирование jina-v4**: Оптимизация работы jina-v4 на доступных GPU (4x 3090). Возможность горизонтального масштабирования при росте нагрузки

- **Кэширование стратегия**: Какие запросы кэшировать? Как долго хранить кэш? Какой размер кэша оптимален?

- **Мультимодальность**: Оптимизация работы с различными комбинациями текста и изображений. Как обрабатывать запросы только с текстом или только с изображением?

- **Выбор векторной БД**: Детальное сравнение ClickHouse vs Elasticsearch для векторного поиска с учетом существующей инфраструктуры и требований к производительности.

- **Мультиязычность**: Поддержка поиска похожих вопросов на разных языках. Требуются ли отдельные модели для каждого языка?

- **Персонализация**: Можно ли улучшить релевантность, учитывая историю запросов пользователя или его предпочтения?

- **Обратная связь**: Механизм сбора feedback от пользователей о релевантности похожих вопросов для улучшения модели.

- [LocalAI для text embeddings в Elasticsearch](https://www.elastic.co/search-labs/blog/localai-for-text-embeddings)
- MTEB Leaderboard для выбора моделей эмбеддингов
