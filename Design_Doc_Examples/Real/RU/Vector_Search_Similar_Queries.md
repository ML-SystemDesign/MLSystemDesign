## I. Problem Definition

### i. Origin

Веб-сервис обрабатывает пользовательские запросы (текст + изображения) с помощью ИИ. Многие пользователи задают похожие или идентичные вопросы, что приводит к:

- **Дублированию запросов к ИИ-сервису**: Установлено, что дубликатов в общем трафике меньше одного процента (~0.6% от снятых эмбеддингов, ~5,000 полных дубликатов). Анализ показал, что дубликаты по системной ошибке составляют ничтожно малую долю от этих 0.6%. При исследовании выявлены случаи, когда пользователи намеренно отправляли идентичные вопросы для получения альтернативных ответов от ИИ. Решение: оставить без изменений, так как количество незначительно и не влияет на работу системы.

- **Высоким затратам на обработку**: Входные токены расходуются в 10 раз больше из-за распознавания изображений на стороне ИИ. Установлено: расход токенов не является приоритетом на текущем этапе, есть резерв для увеличения.

- **Упущенным возможностям для перелинковки и улучшения SEO**: Результат валидации показал высокую эффективность - выявлены семантически идентичные запросы пользователей. Перелинковка = создание связей между запросами через блок "похожие запросы" (вместо блока "другие запросы" где просто выводятся свежие). Легко внедрить после снятия базы эмбеддингов и постройки устойчивого пайплайна обработки.

- **Отсутствию возможности показать пользователю уже существующие ответы**: Нет связи между новыми и существующими вопросами/ответами на сайте.

**Ключевой инсайт**: Перелинковка продемонстрировала высокую эффективность - выявлены семантически идентичные запросы пользователей, что подтверждает ценность решения для создания связей между похожими запросами.

### ii. Relevance & Reasons

**Текущая ситуация**:
- Нет механизма для определения похожих запросов
- Каждый запрос обрабатывается независимо, даже если идентичный вопрос уже был задан ранее
- Нет связи между новыми и существующими вопросами/ответами на сайте
- Расход токенов не является приоритетом на текущем этапе (есть резерв для увеличения)

**Цель**: Создание связей между похожими запросами через блок "похожие запросы" для улучшения SEO и пользовательского опыта.

**Подход**: Оффлайн обработка без дополнительной инфраструктуры, все необходимое доступно on-premises (MongoDB, ClickHouse, GPU 3080/3080Ti).

### iii. Previous Work

Рассмотрены альтернативные подходы:

1. **Self-hosted DeepEncoder + FAISS**: Полный контроль, но требует GPU-инфраструктуры и поддержки
2. **Elasticsearch с LocalAI**: Готовая векторная БД, но требует дополнительной инфраструктуры
3. **ClickHouse с мультимодальными эмбеддингами**: ✅ **Выбрано** - уже в инфраструктуре, единое векторное пространство для текста и изображений
4. **Облачные решения**: Быстрый старт, но постоянные затраты и зависимость от провайдера
5. **CPU-based модели (BLIP, SigLIP)**: Можно запустить без GPU, но высокая latency и проблемы с пропускной способностью

**Выбранное решение**: jina-v4 + ClickHouse
- Предобученная модель jina-v4 для генерации мультимодальных эмбеддингов (текст + изображения) в едином векторном пространстве
- Хранение эмбеддингов прямо в ClickHouse (Array(Float32))
- Все работает локально on-premises, без дополнительной инфраструктуры
- POC уже работает локально, тест на 500k эмбеддингов выполнен

### iv. Other Issues & Risks

**Технические ограничения**:
- Требуется GPU уровня 3080 или выше для production (GPU 3080/3080Ti: ~10 мс до < 200мс, до 4×3080 на хост, пока нет проблем; GPU 3060 Ti: некоторые запросы могут превышать 1 секунду)
- Хранение эмбеддингов в ClickHouse (не специализированная векторная БД, но приемлемо для текущих требований)
- Требования к памяти GPU: ~8 ГБ (находится на границе доступной памяти GPU)
- Поиск в ClickHouse медленнее чем в специализированных векторных БД, но приемлем для текущих требований (2 запроса/сек легко укладываются)

**Бизнес ограничения**:
- Приоритет на оффлайн обработку дубликатов (без дополнительной инфраструктуры)
- Постепенное внедрение решения с вычислением похожих вопросов на on-premises мощностях
- Публикация похожих вопросов на сайте в блоке "похожих" и оценка изменений трафика

## II. Metrics and Losses

### i. Metrics

**Технические метрики**:
- **Latency P95**: < 1 секунда для 95% запросов (целевое значение)
- **Latency P99**: < 1.5 секунды для 99% запросов
- **Precision@5**: > 0.7 (70% похожих вопросов должны быть релевантными)
- **Throughput**: Обработка минимум 2 запросов в секунду (легко укладывается в требования)

**Бизнес метрики** (примерные оценки):
- **Hit Rate**: > 20% пользователей выбирают похожий вопрос вместо отправки своего
- **SEO Impact**: Увеличение внутренних переходов на 10-15% (перелинковка)
- **User Engagement**: Увеличение времени на сайте на 5-10% за счет просмотра похожих вопросов
- **Click-through Rate (CTR)**: > 15% пользователей кликают на похожие вопросы

**Контекст выбора метрик**:
- Проект находится на ранней стадии инновационности - не требуется делать лучший продукт, достаточно применить подходящее решение из хорошо известных и получить value для бизнеса
- Бизнес-метрики выбраны приблизительно ("на глаз"), так как на текущей стадии любое решение уже будет успехом
- Нормальная аналитика по уже опубликованным вопросам - это ongoing задача для бэкенд команды (настройка сбора метрик в ClickHouse)

**Текущие приоритеты**:
- Экономия токенов не является приоритетом на текущем этапе
- Основной фокус: перелинковка и улучшение SEO через публикацию похожих вопросов

### ii. Loss Functions

**Векторный поиск**: cosineDistance в ClickHouse для поиска похожих векторов.

**Особенность**: Вопросы семантически совпадают в топ-5, но по distance могут быть и 0.001, и 0.04, и даже 0.3. Требует дальнейшего исследования для понимания связи между distance и семантическим сходством.

## IV. Validation Schema

### i. Requirements

**Оффлайн валидация (текущая фаза)**: Поиск похожих запросов в индексированных данных для популярных страниц опубликованных вопросов пользователей.

**Онлайн валидация (будущее)**: Realtime поиск с latency < 1s на всех платформах (Web Mobile/PC, Android, iOS).

### ii. Inference Process

**Оффлайн (текущая фаза)**:
1. Генерация мультимодального эмбеддинга через jina-v4 на GPU 3080 для популярных страниц
2. Поиск похожих запросов в ClickHouse через cosineDistance
3. Для популярных страниц предоставление топ-5 по косинусовому сходству
4. Публикация похожих вопросов на сайте в блоке "похожих"
5. Оценка разницы в трафике по метрикам с сайта

**Онлайн (будущее)**:
1. Пользователь вводит запрос (текст + изображения, опционально) в интерфейсе
2. Frontend отправляет мультимодальный запрос в Similarity Search API
3. API отправляет запрос в jina-v4 для генерации мультимодального эмбеддинга (GPU 3080: ~10 мс, GPU 3080Ti: до 200 мс)
4. API выполняет поиск похожих векторов в ClickHouse через cosineDistance по сгенерированному эмбеддингу
5. API получает ID похожих запросов и загружает полные данные из MongoDB
6. API возвращает топ-K похожих запросов с метаданными (текст, изображения, URL)
7. Frontend отображает список похожих вопросов пользователю
8. Пользователь выбирает один из вопросов или отправляет свой запрос на обработку ИИ

### iii. Update Frequency

**Текущая фаза**: Оффлайн индексация запросов из MongoDB через Prefect по типам и периодам:
- ✅ Полностью снят тип 1 (только картинка): 455,450 эмбеддингов
- 🔄 Поставлен на снятие тип 2 (картинка + текст)

**Будущее**: Near-realtime или периодическая индексация новых запросов, периодическая переиндексация при обновлении модели.

## VII. Training Pipeline

### i. Overview

**Архитектура системы**:
- **Embedding Service (jina-v4)**: Генерация мультимодальных эмбеддингов (text + image) в едином векторном пространстве
- **Vector Database (ClickHouse)**: Хранение эмбеддингов в Array(Float32) и поиск по векторным представлениям через cosineDistance
- **MongoDB**: Хранение всех существующих запросов (источник данных: ~3M запросов)
- **Indexing Service (Prefect Orchestration)**: Оффлайн индексация запросов из MongoDB в ClickHouse через Prefect
- **Similarity Search Service**: Поиск похожих запросов (оффлайн для начала, онлайн в будущем)
- **Frontend Integration**: Интеграция с веб и мобильными приложениями (будущее)

**Data Flow (Индексация - Оффлайн, текущая фаза)**:
1. Prefect Flow `embeddings_backfill` запускается по расписанию или вручную с параметрами (start_ts, end_ts, batch_size, model_version)
2. Task `load_next_batch`: Indexing Service извлекает запросы из MongoDB по типам и периодам батчами (paging по времени и/или _id)
3. Task `process_batch` (с тегом `gpu` для concurrency limits): Для каждого батча генерируются мультимодальные эмбеддинги через jina-v4 на GPU 3080
4. Используется готовое представление модели (без модификации векторов)
5. Эмбеддинг + метаданные (ID из MongoDB, текст, изображения, дата, источник) сохраняются в ClickHouse (Array(Float32))
6. Связь между ClickHouse и MongoDB поддерживается через ID запроса
7. Мониторинг через Prefect UI: история запусков, параметры, логи, прогресс (processed/failed/skipped, последний курсор/таймстемп)

### ii. Data Preprocessing

**Предобработка не требуется**: Данные в MongoDB уже находятся в структурированном формате. Каждая запись соответствует pydantic модели AIResponse и представляет собой готовый документ для обработки.

- **Генерация эмбеддингов**: Через jina-v4 на GPU 3080/3080Ti
  - Картинки: время генерации зависит от размера и сложности изображений
  - Текст: обрабатывается с минимальной задержкой
- **Индексация**: В ClickHouse (Array(Float32)) с метаданными и ссылкой на MongoDB ID

### iii. Model Training

**Не требуется** - используем pre-trained модель jina-v4 без дообучения. POC показал отличные результаты на реальных данных.

## IX. Measuring and Reporting

### i. Measuring Results

**Метрики производительности модели**:
- **Precision@K**: Доля релевантных результатов в топ-K (K=5), целевое значение > 0.7
- **Recall@K**: Доля найденных релевантных результатов
- **MRR (Mean Reciprocal Rank)**: Средний обратный ранг первого релевантного результата
- **NDCG@K**: Normalized Discounted Cumulative Gain для оценки качества ранжирования

**Метрики системы**:
- **Latency P50/P95/P99**: Время ответа системы поиска (целевое: P95 < 1s, P99 < 1.5s)
- **Throughput**: Количество запросов в секунду (минимум 2 запроса/сек, легко укладывается)

**Бизнес метрики**:
- **Hit Rate**: Процент запросов, где пользователь выбрал похожий вопрос (целевое: > 20%)
- **SEO Impact**: Увеличение внутренних переходов на 10-15% (перелинковка)
- **User Engagement**: Увеличение времени на сайте на 5-10% за счет просмотра похожих вопросов
- **Click-through Rate (CTR)**: Процент пользователей, кликающих на похожие вопросы (целевое: > 15%)

**Оценка качества**:
- Ручная оценка релевантности на выборке запросов
- A/B тестирование с контрольной группой (без похожих вопросов)
- Мониторинг метрик в production

### ii. A/B Testing

**Планируется после публикации похожих вопросов на сайте**: A/B тестирование (2-4 недели) с сравнением метрик между контрольной и тестовой группами.

### iii. Reporting Results

**Мониторинг**: Prefect UI для индексации, аналитика сайта для бизнес-метрик, real-time дашборды.

**Алерты**: при отклонении latency (P95 < 1s, P99 < 1.5s), снижении качества поиска (Precision@5 < 0.7), ошибках индексации.

**Отчетность**: еженедельные отчеты, итеративное улучшение на основе данных.

## Notes and Learnings

### Key Insights

- **Оффлайн подход**: Эффективен для начала без дополнительной инфраструктуры, используя существующие on-premises ресурсы (MongoDB, ClickHouse, GPU 3080/3080Ti)

- **Производительность jina-v4** (тестирование на машине с Kubernetes, можно запустить разные виртуалки):
  - GPU 3080 (до 4×3080 на хост): Модель демонстрирует высокую производительность - соответствует требованиям < 200 мс, среднее время ~10 мс. Используется для POC и production. Пока нет проблем с производительностью.
  - GPU 3080Ti: Модель показывает производительность до 200 мс на генерацию эмбеддинга (зависит от конкретных картинок). Укладываемся в требования (< 1 секунды).
  - GPU 3060 Ti: Не все запросы, но некоторые могут превышать 1 секунду (> 1000 мс), т.е. могут превышать требования на отдельных запросах, а не типичное значение.
  - **Вывод**: Для production требуется GPU уровня 3080 или выше для гарантированного соблюдения требований по latency.

- **Перелинковка**: Продемонстрировала высокую эффективность - выявлены семантически идентичные запросы пользователей, что подтверждает ценность решения для создания связей между похожими запросами

- **Дубликаты**: Обнаружено ~5,000 полных дубликатов (0.6% от снятых эмбеддингов). Количество посчитано незначительным. Дубликаты по системной ошибке составляют еще меньшую долю от этих 0.6%. При исследовании выявлены интересные случаи, когда пользователи намеренно отправляли идентичные вопросы для получения альтернативных ответов от ИИ.

- **Расход токенов**: Не является приоритетом на текущем этапе. Есть резерв для увеличения. Экономия токенов не является приоритетом на текущем этапе.

- **Постепенное внедрение**: Рекомендуемый подход - вычисление похожих вопросов на on-premises мощностях и публикация их на сайте в блоке "похожих". Оценка изменений трафика по метрикам сайта.

- **Prefect для оркестрации**: Использование Prefect для создания устойчивого пайплайна batch/backfill генерации эмбеддингов с веб-интерфейсом для мониторинга. Архитектура: Prefect Server (self-hosted) + UI, Work Pool, Worker на GPU-VM. Flow `embeddings_backfill` с расписанием и concurrency limits.
