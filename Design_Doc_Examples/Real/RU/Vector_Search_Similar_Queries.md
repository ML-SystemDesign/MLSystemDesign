## I. Problem Definition

### i. Origin

Веб-сервис обрабатывает пользовательские запросы (текст + изображения) с помощью ИИ. Многие пользователи задают похожие или идентичные вопросы, что приводит к:

- Дублированию запросов к ИИ-сервису (установлено: <1% в общем трафике)
- Высоким затратам на обработку (входные токены расходуются в 10 раз больше из-за распознавания изображений)
- Упущенным возможностям для перелинковки и улучшения SEO
- Отсутствию возможности показать пользователю уже существующие ответы

**Ключевой инсайт**: Перелинковка продемонстрировала высокую эффективность - выявлены семантически идентичные запросы пользователей, что подтверждает ценность решения.

**Системная ошибка**: Анализ показал, что дубликатов по системной ошибке ничтожно мало (~0.6%). Решение: оставить без изменений.

### ii. Relevance & Reasons

**Текущая ситуация**:
- Нет механизма для определения похожих запросов
- Каждый запрос обрабатывается независимо
- Нет связи между новыми и существующими вопросами/ответами
- Расход токенов не является приоритетом на текущем этапе (есть резерв для увеличения)

**Цель**: Создание связей между похожими запросами через блок "похожие запросы" для улучшения SEO и пользовательского опыта.

**Подход**: Оффлайн обработка без дополнительной инфраструктуры, все необходимое доступно on-premises.

### iii. Previous Work

Рассмотрены альтернативные подходы:
- Self-hosted DeepEncoder + FAISS
- Elasticsearch с LocalAI
- Облачные решения для инференса
- CPU-based модели (BLIP, SigLIP)

**Выбрано**: jina-v4 + ClickHouse (уже в инфраструктуре, единое векторное пространство для текста и изображений).

### iv. Other Issues & Risks

**Технические ограничения**:
- Требуется GPU уровня 3080 или выше для production
- Хранение эмбеддингов в ClickHouse (не специализированная векторная БД)
- Требования к памяти GPU: ~8 ГБ

**Бизнес ограничения**:
- Приоритет на оффлайн обработку
- Постепенное внедрение решения

## II. Metrics and Losses

### i. Metrics

**Технические метрики**:
- Latency P95: < 1 секунда
- Precision@5: > 0.7
- Throughput: минимум 2 запроса/сек

**Бизнес метрики**:
- Hit Rate: > 20% пользователей выбирают похожий вопрос
- SEO Impact: увеличение внутренних переходов на 10-15%
- User Engagement: увеличение времени на сайте на 5-10%

### ii. Loss Functions

**Векторный поиск**: cosineDistance в ClickHouse для поиска похожих векторов.

## III. Dataset

### i. Data Sources

**MongoDB** - все существующие запросы пользователей.

**Статистика (период: 1 декабря 25 - 1 января 26)**:
- Всего ответов: 3,016,281
- По источникам: Android 36%, iOS 13%, Site 25%
- Типы: Только картинка 74%, Картинка + текст 8%, 2 картинки + текст 0.2%
- Статус индексации: Снято 455,450 эмбеддингов (тип 1 полностью), дубликатов ~5,000 (0.6%)

### ii. Data Labeling

Не требуется - используем существующие запросы.

### iii. Available Metadata

ID запроса, текст, изображения, дата создания, статус, источник, URL.

### iv. Data Quality Issues

Дубликаты незначительны и не требуют обработки.

### v. ETL Pipeline

**Prefect Orchestration**:
- Извлечение запросов из MongoDB по типам и периодам
- Генерация мультимодальных эмбеддингов через jina-v4 на GPU 3080
- Загрузка в ClickHouse (Array(Float32))
- Мониторинг через веб-интерфейс Prefect

## IV. Validation Schema

### i. Requirements

**Оффлайн валидация**: Поиск похожих запросов в индексированных данных.

**Онлайн валидация (будущее)**: Realtime поиск с latency < 1s.

1. Генерация мультимодального эмбеддинга через jina-v4
2. Поиск похожих векторов через cosineDistance в ClickHouse
3. Возврат топ-K похожих запросов

### iii. Update Frequency

**Текущая фаза**: Оффлайн индексация по типам запросов.

**Будущее**: Near-realtime или периодическая индексация новых запросов.

## V. Baseline Solution

### i. Constant Baseline

Текущая система без механизма поиска похожих запросов.

### ii. Model Baselines

**Выбранная модель: jina-v4**
- Мультимодальные эмбеддинги (текст + изображения) в едином пространстве
- Производительность на GPU 3080: среднее время ~10 мс, соответствует требованиям < 200 мс
- Производительность на GPU 3080Ti: до 200 мс
- GPU 3060 Ti: не соответствует требованиям (> 1000 мс на некоторых запросах)

**Альтернативы (рассматривались)**: CLIP, SigLIP модели.

### iii. Feature Baselines

Готовое представление модели jina-v4 без модификации векторов.

## VI. Error Analysis

### i. Learning Curve Analysis

POC показал высокое семантическое сходство для похожих вопросов.

**Особенность**: Вопросы семантически совпадают в топ-5, но по distance могут быть 0.001, 0.04, и даже 0.3. Требует дальнейшего исследования.

### ii. Residual Analysis

Дубликаты незначительны (~0.6%), не требуют обработки.

### iii. Best/Worst Case Analysis

**Лучшие случаи**: Выявлены идентичные документы в различных условиях.

**Худшие случаи**: GPU 3060 Ti не справляется с некоторыми запросами - требуется более мощная GPU.

## VII. Training Pipeline

### i. Overview

**Архитектура**:
- Embedding Service (jina-v4) - генерация эмбеддингов
- Vector Database (ClickHouse) - хранение и поиск
- MongoDB - источник данных
- Indexing Service (Prefect) - оркестрация индексации
- Similarity Search Service - поиск похожих запросов

### ii. Data Preprocessing

- Очистка текста (удаление HTML, нормализация)
- Предобработка изображений (resize, нормализация)
- Генерация эмбеддингов через jina-v4

### iii. Model Training

Не требуется - используем pre-trained модель jina-v4.

### iv. Experiment Tracking

Мониторинг через Prefect UI.

## Testing Results

**Статистика**:

**Качество поиска**:
- Выявлены идентичные документы в различных условиях

### Key Insights

- **Дубликаты**: Незначительны (~0.6%), не требуют обработки
- **Расход токенов**: Не является приоритетом на текущем этапе
